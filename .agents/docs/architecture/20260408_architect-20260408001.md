### 1.2 数据流时序图

```
解锁流程:
用户输入密码 → LockScreen.vue → verifyPassword() → vaultStore.unlock()
  → vaultStore.startIdleDetection() → 从DB加载autoLockTimeout → 启动定时器

用户活动流程:
用户活动 → App.vue捕获事件 → throttle节流 → vaultStore.resetIdleTimer()
  → 清除旧定时器 → 设置新定时器(autoLockTimeout分钟后执行lock())

自动锁定流程:
空闲超时 → vaultStore执行lock() → 清除敏感数据 → isLocked=true
  → App.vue切换显示LockScreen → stopIdleDetection() → 清除事件监听器
```

## 三、技术选型

### 3.1 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 空闲检测位置 | 渲染进程 (Vue) | 实现简单，可精确检测应用内活动，符合需求约束 |
| 计时器管理 | Pinia Store | 状态集中管理，与锁定逻辑解耦 |
| 事件监听位置 | App.vue | 全局根组件，生命周期管理清晰 |
| 节流策略 | lodash.throttle 或自定义 | mousemove高频事件需要节流，避免性能影响 |
| 定时器类型 | setTimeout | 简单直接，配合clearTimeout重置 |

### 3.2 为什么不使用 Electron powerMonitor

需求约束明确规定使用前端渲染进程实现，不依赖主进程powerMonitor。原因：

1. **实现范围**：当前版本仅检测应用内活动，不需要系统级别空闲检测
2. **跨平台一致性**：前端实现不依赖Electron版本差异
3. **简单可控**：渲染进程实现更简单，便于调试和维护

**预留扩展**：后续版本如需系统级别检测，可引入主进程powerMonitor，通过IPC通知渲染进程。

## 四、详细设计

### 4.1 vault.ts 新增状态和方法

```typescript
// 新增状态
const autoLockTimeout = ref<number>(5)  // 默认5分钟
const idleTimer = ref<number | null>(null)
const isIdleDetectionActive = ref<boolean>(false)

// 新增方法
async function loadAutoLockTimeout(): Promise<void> {
  const settings = await window.electronAPI.db.getSettings()
  autoLockTimeout.value = settings.autoLockTimeout ?? 5
}

function startIdleDetection(): void {
  if (isIdleDetectionActive.value) return
  
  isIdleDetectionActive.value = true
  resetIdleTimer()
}

function stopIdleDetection(): void {
  if (idleTimer.value !== null) {
    clearTimeout(idleTimer.value)
    idleTimer.value = null
  }
  isIdleDetectionActive.value = false
}

function resetIdleTimer(): void {
  if (!isIdleDetectionActive.value) return
  
  // 清除旧定时器
  if (idleTimer.value !== null) {
    clearTimeout(idleTimer.value)
  }
  
  // 设置新定时器
  const timeoutMs = autoLockTimeout.value * 60 * 1000
  idleTimer.value = window.setTimeout(() => {
    lock()  // 空闲超时，自动锁定
  }, timeoutMs)
}

// 修改现有lock方法
function lock() {
  stopIdleDetection()  // 锁定后停止检测
  // ... 现有逻辑保持不变
}

// 修改现有unlock方法
function unlock(key: string) {
  masterKey.value = key
  isLocked.value = false
  startIdleDetection()  // 解锁后启动检测
}
```

### 4.2 App.vue 事件监听设计

```typescript
<script setup lang="ts">
import { onMounted, onUnmounted, watch } from 'vue'
import { useVaultStore } from './stores/vault'

const vaultStore = useVaultStore()

// 节流函数
function throttle(fn: Function, delay: number) {
  let lastCall = 0
  return function(...args: any[]) {
    const now = Date.now()
    if (now - lastCall >= delay) {
      lastCall = now
      fn.apply(this, args)
    }
  }
}

// 节流处理后的重置函数 (mousemove 150ms节流)
const throttledReset = throttle(() => {
  vaultStore.resetIdleTimer()
}, 150)

// 事件处理器
function handleUserActivity(event: string) {
  if (!vaultStore.isIdleDetectionActive) return
  
  // mousemove已节流，其他事件直接调用
  if (event === 'mousemove') {
    throttledReset()
  } else {
    vaultStore.resetIdleTimer()
  }
}

// 生命周期管理
onMounted(() => {
  window.addEventListener('mousemove', () => handleUserActivity('mousemove'))
  window.addEventListener('keydown', () => handleUserActivity('keydown'))
  window.addEventListener('click', () => handleUserActivity('click'))
})

onUnmounted(() => {
  window.removeEventListener('mousemove', () => handleUserActivity('mousemove'))
  window.removeEventListener('keydown', () => handleUserActivity('keydown'))
  window.removeEventListener('click', () => handleUserActivity('click'))
})

// 监听锁定状态，管理事件监听器有效性
watch(() => vaultStore.isLocked, (isLocked) => {
  if (isLocked) {
    // 已锁定，事件监听器不会触发任何操作（isIdleDetectionActive为false）
  }
})
</script>
```

### 4.3 LockScreen.vue 微调

解锁成功后需要确保空闲检测正确启动。由于 `vaultStore.unlock()` 已集成 `startIdleDetection()`，无需额外修改。

```typescript
// LockScreen.vue handleUnlock()
async function handleUnlock() {
  // ... 验证逻辑
  if (valid) {
    vaultStore.unlock(password.value)  // unlock已包含startIdleDetection

  }
}
```

**注意**：需要在解锁后调用 `loadAutoLockTimeout()` 确保配置已加载。可以在应用初始化时预加载。

## 五、性能优化策略

### 5.1 事件节流

| 事件类型 | 频率特性 | 节流策略 |
|----------|----------|----------|
| mousemove | 高频 (~60fps) | 150ms 节流 |
| keydown | 中频 | 不节流（直接调用） |
| click | 低频 | 不节流（直接调用） |

### 5.2 定时器管理

- 使用 `setTimeout` + `clearTimeout` 组合，每次重置清除旧定时器
- 避免定时器泄漏：`lock()` 和 `stopIdleDetection()` 都会清理
- 使用 `window.setTimeout` 返回的ID，避免类型混淆

### 5.3 内存管理

- 锁定后停止检测，清除所有定时器和事件监听器引用
- `onUnmounted` 确保组件销毁时清理资源

## 六、安全架构考虑

### 6.1 数据隔离

- `autoLockTimeout` 配置存储在主进程数据库
- 渲染进程通过IPC获取配置值，不直接操作数据库
- 符合现有安全架构原则

### 6.2 锁定触发安全性

- `lock()` 方法已实现：
  - 清除剪贴板中的密码数据
  - 清除内存中的主密钥
  - 清除内存中的密码条目
  - 重置所有状态
- 空闲检测触发 `lock()` 后，所有敏感数据被彻底清除

### 6.3 防绕过

- 空闲检测在渲染进程实现，用户无法通过修改数据库绕过
- 定时器在内存中运行，无法被持久化篡改

## 七、错误处理

### 7.1 配置加载失败

```typescript
async function loadAutoLockTimeout(): Promise<void> {
  try {
    const settings = await window.electronAPI.db.getSettings()
    autoLockTimeout.value = settings.autoLockTimeout ?? 5
  } catch (error) {
    console.error('Failed to load autoLockTimeout:', error)
    autoLockTimeout.value = 5  // 使用默认值
  }
}
```

### 7.2 IPC异常

- 如果 `db.getSettings()` 失败，使用默认值5分钟
- 不影响应用正常运行，静默降级
