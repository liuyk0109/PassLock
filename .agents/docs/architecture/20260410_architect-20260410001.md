## 2. 模块划分

### 2.1 主进程模块变更

#### crypto.ts（无需修改）
现有API完全满足需求：

#### database.ts（无需修改）
现有API完全满足需求：
- `getEntries()` - 获取所有条目
- `setEntries(entries)` - 批量更新条目
- `getMasterKeyVerify()` - 获取验证数据
- `setMasterKeyVerify(verify)` - 更新验证数据

#### main.ts（新增IPC Handler）
新增一个复合操作IPC handler，在主进程中完成所有重加密逻辑：

```typescript
ipcMain.handle('crypto:changeMasterPassword',
  async (event, currentPassword: string, newPassword: string) => {
    // 1. 验证当前密码
    // 2. 备份当前数据到内存
    // 3. 逐条解密+重新加密，发送进度事件
    // 4. 更新验证数据
    // 5. 失败时自动回滚
    // 6. 清除内存备份
    return { success: true }
  }
)
```

**设计理由**：
- 主密码和加密操作只在主进程执行，渲染进程不接触明文密码
- 备份和回滚在主进程内存中完成，安全且高效
- 进度反馈通过 `event.sender.send()` 推送

#### preload.ts（新增API暴露）

```typescript
// 新增 crypto 命名空间下
changeMasterPassword: (
  currentPassword: string,
  newPassword: string,
  onProgress?: (current: number, total: number) => void
) => Promise<{ success: boolean; error?: string }>
```

**实现方式**：使用 `ipcRenderer.invoke` + `ipcRenderer.on` 监听进度事件

## 3. 数据流设计

### 3.1 密码修改完整数据流

```
用户点击"确认修改"
    │
    ▼
ChangePasswordModal 验证表单
  - 当前密码非空
  - 新密码强度 >= medium (40分)
  - 确认密码匹配
    │
    ▼
vaultStore.changeMasterPassword(currentPwd, newPwd, onProgress)
    │
    ▼
electronAPI.crypto.changeMasterPassword(currentPwd, newPwd, onProgress)
    │  ──── IPC invoke ────▶
    │
    │                    主进程 handler:
    │                    1. verifyPassword(verifyData, currentPwd)
    │                       ├─ 失败 → return { success: false, error: '当前密码错误' }
    │                       └─ 成功 ↓
    │                    2. backup = { entries: [...getEntries()], verify: getMasterKeyVerify() }
    │                    3. for (entry of entries):
    │                       plaintext = decrypt(entry.password, currentPwd)
    │                       entry.password = encrypt(plaintext, newPassword)
    │                       event.sender.send('change-password-progress', i+1, total)
    │                    4. setEntries(reEncryptedEntries)
    │                    5. newVerify = createVerifyData(newPassword)
    │                    6. setMasterKeyVerify(newVerify)
    │                    7. 清除 backup 内存引用
    │                    ◀── IPC result ────
    │
    ▼
vaultStore 更新状态:
  - masterKey.value = newPassword
  - entries.value = reEncryptedEntries (从数据库reload)
  - 保持解锁状态
    │
    ▼
ChangePasswordModal 显示成功 → 关闭弹窗 → Toast提示
```

### 3.2 备份回滚数据流

```
主进程 changeMasterPassword handler 内部:

try {
  // Step 1: 验证
  verifyPassword(verifyData, currentPassword)  // 失败直接return

  // Step 2: 内存快照备份
  const backup = {
    entries: JSON.parse(JSON.stringify(getEntries())),  // 深拷贝
    masterKeyVerify: getMasterKeyVerify()
  }

  // Step 3: 逐条重加密 (可能失败)
  for (let i = 0; i < entries.length; i++) {
    // decrypt可能抛出异常
    const plaintext = decrypt(entries[i].password, currentPassword)
    entries[i].password = encrypt(plaintext, newPassword)
    // 发送进度
    event.sender.send('change-password-progress', i + 1, entries.length)
  }

  // Step 4: 写入数据库 (可能失败)
  await setEntries(entries)
  const newVerify = createVerifyData(newPassword)
  await setMasterKeyVerify(newVerify)

} catch (error) {
  // 回滚: 恢复备份
  if (backup) {
    await setEntries(backup.entries)
    await setMasterKeyVerify(backup.masterKeyVerify)
  }
  return { success: false, error: error.message }
} finally {
  // 清除内存备份 (安全)
  backup = null
}
```

## 4. IPC通信协议

### 4.1 新增IPC通道

| 通道名 | 方向 | 参数 | 返回值 | 说明 |
|--------|------|------|--------|------|
| `crypto:changeMasterPassword` | 渲染→主 | `currentPassword: string, newPassword: string` | `{ success: boolean; error?: string }` | 主密码修改（含验证、备份、重加密、回滚） |
| `change-password-progress` | 主→渲染 | `current: number, total: number` | - | 加密进度推送 |

### 4.2 现有IPC通道复用

无需新增其他IPC通道，所有操作封装在 `crypto:changeMasterPassword` 中完成。

### 4.3 Preload API 设计

```typescript
// electronAPI.crypto 新增
changeMasterPassword: (
  currentPassword: string,
  newPassword: string,
  onProgress?: (current: number, total: number) => void
) => Promise<{ success: boolean; error?: string }>
```

**进度回调实现**：

```typescript
changeMasterPassword: (currentPassword, newPassword, onProgress) => {
  // 注册进度监听
  const progressChannel = 'change-password-progress'
  if (onProgress) {
    const handler = (_event: any, current: number, total: number) => {
      onProgress(current, total)
    }
    ipcRenderer.on(progressChannel, handler)
  }

  return ipcRenderer.invoke('crypto:changeMasterPassword', currentPassword, newPassword)
    .finally(() => {
      if (onProgress) {
        ipcRenderer.removeAllListeners(progressChannel)
      }
    })
}
```

## 6. 安全架构

### 6.1 敏感数据隔离

| 数据 | 存储位置 | 传输路径 | 清理策略 |
|------|----------|----------|----------|
| 当前主密码 | 仅渲染进程内存 | 渲染→主进程IPC | 函数调用结束即GC |
| 新主密码 | 仅渲染进程内存 | 渲染→主进程IPC | 函数调用结束即GC |
| 明文密码条目 | 仅主进程内存 | 不离开主进程 | 变量作用域结束即GC |
| 内存备份 | 仅主进程内存 | 不传输 | finally块中置null |
| 加密后条目 | 数据库 + 渲染进程 | 主进程→数据库→渲染 | 正常GC |

### 6.2 安全保障措施

1. **密码不落盘**：备份数据仅存内存，不写入临时文件
2. **即时清理**：finally块确保备份引用置null，避免内存驻留
3. **原子操作**：验证→备份→重加密→更新，任一环节失败自动回滚
4. **强度门槛**：渲染进程预验证强度>=40分，防止弱密码提交到主进程
5. **进程隔离**：contextBridge确保IPC通信安全，无nodeIntegration暴露

### 6.3 备份回滚策略

- **备份方式**：`JSON.parse(JSON.stringify(getEntries()))` 深拷贝entries + masterKeyVerify
- **备份存储**：主进程局部变量，handler返回后自动GC
- **回滚触发**：try-catch捕获任何异常（decrypt失败、encrypt失败、DB写入失败）
- **回滚操作**：`setEntries(backup.entries)` + `setMasterKeyVerify(backup.masterKeyVerify)`
- **清理保障**：finally块清除备份引用，无论成功失败

## 8. 错误处理策略

### 8.1 错误分类

| 错误类型 | 发生位置 | 处理策略 | 用户提示 |
|----------|----------|----------|----------|
| 当前密码错误 | 主进程验证 | 直接返回错误 | "当前密码错误" |
| 解密条目失败 | 主进程重加密 | 回滚+返回错误 | "加密失败，数据已恢复" |
| 加密条目失败 | 主进程重加密 | 回滚+返回错误 | "加密失败，数据已恢复" |
| 数据库写入失败 | 主进程持久化 | 回滚+返回错误 | "保存失败，数据已恢复" |
| IPC通信失败 | 渲染进程 | 异常捕获 | "操作失败，请重试" |
| 新密码强度不足 | 渲染进程 | 表单验证 | "新密码强度需达到中等以上" |

### 8.2 恢复保障

- 所有可预见错误均有回滚机制
- 回滚操作本身若失败，记录错误日志但不影响用户（原数据在数据库层面可能已部分修改，需手动介入）
- 渲染进程catch所有异常，确保UI不崩溃

## 9. 性能设计

### 9.1 加密性能

- AES-256-GCM 单次加密 < 1ms（Node.js crypto模块）
- 100条目重加密预计 < 100ms
- 1000条目重加密预计 < 1s
- PBKDF2密钥派生是瓶颈：100000次迭代约100ms/密码

### 9.2 进度反馈优化

- 每条目加密完成后推送进度事件
- 渲染进程通过 `requestAnimationFrame` 节流UI更新
- 进度条使用CSS transition动画平滑过渡

### 9.3 内存优化

- 备份数据为JSON深拷贝，仅占用必要内存
- 不使用Buffer暂存明文密码，逐条处理
- finally块立即释放备份引用

## 12. ADR - 架构决策记录

### ADR-001: 重加密逻辑放在主进程而非渲染进程

**上下文**：密码修改需要逐条解密再重新加密，可选择在渲染进程或主进程执行。

**决策**：在主进程中通过单一IPC handler完成所有重加密逻辑。

**理由**：
- 主密码永远不应离开主进程（安全原则）
- 渲染进程仅传递currentPassword和newPassword，主进程负责实际解密/加密
- 备份和回滚在主进程内存中完成，减少IPC调用次数
- 减少明文密码在进程间传输的风险

**后果**：
- 主进程handler逻辑较复杂
- 进度反馈需要额外IPC事件通道
- 渲染进程无法直接控制重加密流程

### ADR-002: 备份使用内存快照而非文件

**上下文**：修改前需要备份原数据，可选择临时文件或内存存储。

**决策**：使用主进程内存深拷贝作为备份。

**理由**：
- 无文件残留风险（临时文件可能被恢复）
- GC自动回收，无需手动清理文件
- 性能好，无需磁盘I/O
- 数据量通常不大（条目数通常<1000）

**后果**：
- 极大量条目时内存占用增加（可接受）
- 进程崩溃时备份丢失（但数据库可能已写入部分数据，需手动介入）

### ADR-003: 页面路由使用状态切换而非vue-router

**上下文**：设置页面需要独立的页面路由，可选择vue-router或状态切换。

**决策**：使用vaultStore.currentPage状态切换，与现有App.vue架构一致。

**理由**：
- 现有架构已使用 `v-if="isLocked"` 切换LockScreen/VaultPage
- 引入vue-router增加依赖和复杂度
- 当前只需vault/settings两个页面，状态切换足够
- Transition动画可复用现有page动画

**后果**：
- 未来页面增多时可能需要重构为vue-router
- URL不会变化，无法直接访问/settings

### ADR-004: 单一IPC handler封装完整修改流程

**上下文**：密码修改涉及验证、备份、重加密、更新验证数据等步骤，可拆分为多个IPC调用或在单个handler中完成。

**决策**：使用单个 `crypto:changeMasterPassword` handler封装完整流程。

**理由**：
- 原子性保障：所有步骤在同一个handler中完成，避免中间状态
- 减少IPC调用次数，降低通信开销
- 备份和回滚逻辑不需要跨IPC调用协调
- 进度反馈通过事件推送，不影响handler的主流程

**后果**：
- handler逻辑较长，需注意代码组织
- 渲染进程对流程的控制粒度较粗
