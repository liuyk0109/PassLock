## 二、AES-256-GCM 密钥轮换安全分析

### 2.1 IV唯一性保证

AES-GCM 最关键的安全约束：**同一密钥不得重复使用相同IV**。

**密钥轮换场景分析**：

1. **旧密钥空间与新密钥空间完全独立**
   - 新密码 → 新随机salt(32字节) → PBKDF2 → 全新256位密钥
   - 即使新密钥碰巧使用与旧密钥相同的IV，也完全安全
   - NIST SP 800-38D: 不同密钥的IV空间互不影响

2. **同一次轮换内的IV安全性**
   - `encrypt()` 每次调用生成**独立随机12字节IV**
   - IV碰撞概率 = 1/2^96 ≈ 1.3×10^-29，远低于安全阈值(2^-32)
   - 即使加密1000条目，碰撞概率仍可忽略

3. **轮换前后的IV独立性**
   - 旧密钥的IV使用记录不影响新密钥
   - 新密钥的IV空间是"干净的"——无需追踪旧IV

**结论：密钥轮换天然消除了IV重用风险，无需任何额外处理。**

### 2.2 Salt刷新机制

当前 `encrypt()` 实现每次生成新salt：

```
encrypt(plaintext, newPassword) 内部流程：
  salt = randomBytes(32)      ← 每次随机
  iv = randomBytes(12)         ← 每次随机
  key = PBKDF2(newPassword, salt, 100000, 32, 'sha256')
  cipher = AES-256-GCM(key, iv)
  ...
```

**安全效果**：
- 每个条目使用**独立的salt + IV**，派生出**不同的密钥**
- 即使用同一密码加密相同明文，密文也完全不同
- 重加密后的条目与旧条目无任何关联信息可被利用

### 2.3 旧格式自动迁移

现有 `decrypt()` 已支持双格式IV自动检测（12字节/16字节）：

```typescript
// decrypt 内部逻辑（已有）：
// 1. 先尝试12字节IV（新格式）
// 2. 失败则回退16字节IV（旧格式）
```

**密钥轮换的额外收益**：

| 特性 | 轮换前 | 轮换后 |
|------|--------|--------|
| IV长度 | 可能混合12/16字节 | 统一为12字节 |
| Salt | 旧salt | 全新salt |
| 密钥派生 | 旧密钥 | 全新密钥 |
| 密文格式 | 可能混合新旧格式 | 统一新格式 |

**密钥轮换是IV格式迁移的天然契机**——轮换完成后，所有条目自动升级为12字节IV新格式。

### 2.4 密钥独立性

密钥轮换确保新旧密钥完全独立：

```
旧密钥派生: PBKDF2(oldPassword, oldSalt_1, 100000) → key_old_1
旧密钥派生: PBKDF2(oldPassword, oldSalt_2, 100000) → key_old_2
                           ↕ 完全独立
新密钥派生: PBKDF2(newPassword, newSalt_1, 100000) → key_new_1
新密钥派生: PBKDF2(newPassword, newSalt_2, 100000) → key_new_2
```

- 不同密码 + 不同salt = 数学上保证派生不同的密钥
- 即使新旧密码相同（用户改回原密码），不同的salt也确保密钥不同
- **前向安全性**：新密码泄露不影响旧密钥加密的数据（旧salt已不存在）

## 三、重加密流程密码学规范

### 3.1 主进程Handler内的密码学操作

```typescript
// crypto:changeMasterPassword handler 内部流程

// Step 1: 验证当前密码（密码学验证）
const verifyData = getMasterKeyVerify()  // 获取加密的验证数据
const isValid = verifyPassword(verifyData, currentPassword)
// verifyPassword 内部: decrypt(verifyData, password) 成功即验证通过
// 验证数据内容: AES-256-GCM加密的 'PASSLOCK_VERIFY' 字符串

// Step 2: 逐条重加密
for (let i = 0; i < entries.length; i++) {
  // 2a: 用旧密码解密
  const plaintext = decrypt(entries[i].password, currentPassword)
  // decrypt 内部:
  //   - 从密文提取 salt_old, iv_old, authTag_old, ciphertext
  //   - PBKDF2(currentPassword, salt_old) → key_old
  //   - AES-256-GCM-Decrypt(key_old, iv_old, authTag_old, ciphertext) → plaintext
  //   - GCM认证标签验证确保密文完整性

  // 2b: 用新密码加密
  entries[i].password = encrypt(plaintext, newPassword)
  // encrypt 内部:
  //   - 生成新 salt_new (32字节随机)
  //   - 生成新 iv_new (12字节随机)
  //   - PBKDF2(newPassword, salt_new) → key_new
  //   - AES-256-GCM-Encrypt(key_new, iv_new, plaintext) → {authTag, ciphertext}
  //   - 输出: base64(salt_new + iv_new + authTag + ciphertext)
}

// Step 3: 更新验证数据
const newVerifyData = createVerifyData(newPassword)
// createVerifyData 内部:
//   encrypt('PASSLOCK_VERIFY', newPassword)
//   → 新的salt + iv + authTag + 密文
```

### 3.2 密码学安全保障总结

| 保障项 | 实现机制 | 安全级别 |
|--------|----------|----------|
| 密钥派生安全性 | PBKDF2-SHA256, 100K迭代 | ✅ NIST推荐 |
| 加密算法安全性 | AES-256-GCM | ✅ AEAD认证加密 |
| IV唯一性 | 每次随机12字节 | ✅ 碰撞概率2^-96 |
| Salt唯一性 | 每次随机32字节 | ✅ 碰撞概率2^-256 |
| 数据完整性 | GCM认证标签(16字节) | ✅ 防篡改 |
| 密钥独立性 | 不同密码+不同salt | ✅ 数学保证 |
| 前向安全性 | 旧salt随重加密消失 | ✅ 旧密文不可恢复 |

## 四、性能分析与优化建议

### 4.1 当前方案性能特征

每次 `encrypt()` 调用执行一次完整的PBKDF2派生（100,000次迭代）：

| 条目数 | PBKDF2次数 | 预估耗时 | 用户体验 |
|--------|-----------|----------|----------|
| 10 | 10 | ~1秒 | 即时完成 |
| 100 | 100 | ~10秒 | 需进度反馈 ✅ |
| 500 | 500 | ~50秒 | 需进度反馈 ✅ |
| 1000 | 1000 | ~100秒 | 较长等待 |

**PBKDF2是性能瓶颈**，占总耗时99%以上。

### 4.2 优化方案（可选，非当前任务必需）

**方案A：单次派生 + 多IV加密**

```typescript
// 新增底层函数（仅主进程内部使用）
function encryptWithKey(plaintext: string, key: Buffer): string {
  const iv = randomBytes(IV_LENGTH)
  const cipher = createCipheriv(ALGORITHM, key, iv, { authTagLength: AUTH_TAG_LENGTH })
  const ciphertext = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
  const authTag = cipher.getAuthTag()
  // 格式: iv + authTag + ciphertext（无salt，因为密钥已派生）
  return Buffer.concat([iv, authTag, ciphertext]).toString('base64')
}
```

优化后的轮换流程：
1. `deriveKey(newPassword, newSalt)` — 仅1次PBKDF2
2. 所有条目使用同一密钥 + 不同IV加密
3. 验证数据仍使用完整 `encrypt()` （含独立salt）

| 条目数 | 优化前 | 优化后 | 提升 |
|--------|--------|--------|------|
| 100 | ~10秒 | ~0.2秒 | 50x |
| 1000 | ~100秒 | ~1秒 | 100x |

**安全影响**：同一密钥+不同IV，在12字节IV空间下碰撞概率2^-96，仍然安全。但需注意：
- 密钥仅保护当前条目集，泄露该密钥影响所有条目
- 不如每条目独立salt+密钥的"深度防御"
- 验证数据需保持独立salt+密钥派生

**建议**：当前阶段保持现有方案（架构已设计进度反馈覆盖），未来条目数超过500时可考虑优化。

### 4.3 解密性能注意

`decrypt()` 函数对旧格式（16字节IV）的兼容逻辑包含try-catch回退：

```typescript
// 先尝试12字节IV → 失败则回退16字节IV
```

这意味着旧格式条目解密时会多执行一次失败的解密尝试，额外开销约100ms（PBKDF2）。密钥轮换完成后，所有条目自动迁移为新格式，此开销消失。

## 五、安全考量

### 5.1 已知威胁及对策

| 威胁 | 影响 | 对策 | 状态 |
|------|------|------|------|
| 中间条目加密失败 | 部分条目用新密钥，部分用旧密钥 | 内存备份+自动回滚 | ✅ 架构已设计 |
| 数据库写入失败 | 内存中已重加密但未持久化 | 内存备份+自动回滚 | ✅ 架构已设计 |
| 进程崩溃 | 备份丢失，数据库可能部分写入 | 无法自动恢复（可接受） | ⚠️ 已知限制 |
| 旧密码暴力破解 | 旧密文已被覆盖，无法获取 | 重加密后旧密文消失 | ✅ 天然防护 |
| 重加密时内存dump | 明文密码短暂存在于内存 | 主进程安全隔离 | ✅ 可接受 |

### 5.2 进程崩溃场景

**风险**：重加密过程中进程崩溃（如断电），可能出现：
- 数据库已部分写入重加密条目
- 内存备份已丢失
- 无法自动恢复

**缓解措施**：
- 架构设计中 `setEntries()` 是**原子写入**（覆盖整个entries数组），不会出现"半新半旧"状态
- 真正的崩溃风险点是 `setEntries()` 成功但 `setMasterKeyVerify()` 未执行
- **建议**：在handler中确保先写入entries，再写入verify数据（当前设计已是此顺序）

**数据一致性**：
- 若 `setEntries` 成功 + `setMasterKeyVerify` 未执行 → 验证数据仍为旧密码，用户仍可用旧密码解锁
- 此时entries已用新密码加密 → **旧密码解锁后无法解密条目** → 数据不一致

**增强建议**（可选）：将entries和verify数据的写入合并为单次数据库操作：

```typescript
// 在 database.ts 中新增
async function atomicUpdateEntriesAndVerify(
  entries: VaultEntry[],
  verify: string
): Promise<void> {
  const database = getDatabase()
  database.data.entries = entries
  database.data.masterKeyVerify = verify
  await database.write()  // 单次写入
}
```

### 5.3 安全编码要求

主进程handler实现时必须遵守：

1. **明文密码变量及时清理**
   ```typescript
   let plaintext = decrypt(entry.password, currentPassword)
   entry.password = encrypt(plaintext, newPassword)
   plaintext = ''  // 虽然V8 GC不可控，但至少清除引用
   ```

2. **备份数据finally清理**
   ```typescript
   finally {
     backup = null  // 必须在finally中清理
   }
   ```

3. **错误信息不得泄露加密细节**
   - 不暴露salt值、IV值、密钥派生参数
   - 不暴露具体哪条目加密失败
   - 仅返回用户可理解的错误信息

4. **进度事件不包含明文**
