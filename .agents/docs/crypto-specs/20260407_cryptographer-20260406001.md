## 一、现有加密实现审查

### 1.1 算法配置现状 (`electron/crypto.ts`)

| 参数 | 当前值 | NIST推荐 | 评估 |
|------|--------|----------|------|
| 加密算法 | AES-256-GCM | AES-256-GCM | ✅ 符合规范 |
| 密钥派生 | PBKDF2 | PBKDF2/Argon2 | ✅ 符合规范 |
| PBKDF2迭代 | 100,000 | ≥100,000 | ✅ 符合规范 |
| 盐值长度 | 32字节 | ≥16字节 | ✅ 充足 |

| 认证标签 | GCM内置 | 16字节 | ✅ 符合规范 |

### 1.2 发现问题

**P1-IV长度问题**：
- AES-GCM 标准推荐 IV 为 12 字节（96位）
- NIST SP 800-38D Section 8.2.1 明确指出：IV长度应为12字节
- 当前 16 字节 IV 在 Node.js crypto 中可正常工作，但：
  - 性能略有下降（内部需要额外处理）
  - 与标准规范不一致，可能影响与其他系统的兼容性

## 二、算法选择建议

### 2.1 密钥派生算法

**推荐**: PBKDF2（保持现状）

**参数配置**：
```typescript
const PBKDF2_ITERATIONS = 100000  // 满足当前安全标准
const KEY_LENGTH = 32             // 256 bits
const SALT_LENGTH = 32            // bytes
const HASH_ALGORITHM = 'sha256'   // HMAC-SHA256
```

**理由**：
- PBKDF2 是 NIST 推荐的密钥派生函数
- 100,000 次迭代在当前硬件环境下提供足够的安全性
- Argon2 更抗 GPU/ASIC 攻击，但 PBKDF2 在桌面应用场景足够

### 2.2 对称加密算法

**推荐**: AES-256-GCM（保持现状，修正IV长度）

**参数配置**：
```typescript
const ALGORITHM = 'aes-256-gcm'
const IV_LENGTH = 12      // ⚠️ 修正：从16改为12字节
const AUTH_TAG_LENGTH = 16
```

**修正影响评估**：
- **向后兼容性**: 需要处理旧数据（16字节IV）的解密
- **新数据格式**: 新加密数据使用12字节IV
- **迁移策略**: 建议渐进式迁移（见第四章）

### 2.3 哈希算法

**现状**: SHA-256（密码强度评估、验证数据）

**评估**: ✅ 符合规范，无需修改

## 三、密钥管理方案

### 3.1 主密钥生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                    主密钥生命周期                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户输入主密码                                              │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────────┐                                            │
│  │ PBKDF2派生  │ ← 验证数据解密成功                          │
│  │ masterKey   │                                            │
│  └─────────────┘                                            │
│       │                                                     │
│       ▼                                                     │
│  存储位置: 渲染进程 VaultStore.masterKey (ref<string>)       │
│       │                                                     │
│       ├──────────────────────┐                              │
│       │                      │                              │
│       ▼                      ▼                              │
│  IPC调用加密            IPC调用解密                          │
│  (传递masterKey)        (传递masterKey)                     │
│       │                      │                              │
│       ▼                      ▼                              │
│  主进程执行            主进程执行                            │
│  AES-GCM加密          AES-GCM解密                           │
│                                                             │
│  锁定密码库时: masterKey.value = null                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 密钥传递机制分析

**当前实现**: 每次IPC调用都传递 masterKey 字符串

**架构声明**: "主密码永不离开渲染进程传递给主进程"

**⚠️ 声明与实现不一致**

**两种改进方案**：

| 方案 | 描述 | 安全性 | 改动范围 |
|------|------|--------|----------|
| **A. Session模式** | 主进程维护加密session，渲染进程发送请求ID | 🔒 更安全 | 需重构IPC架构 |
| **B. 当前模式** | 保持现状，修正架构文档声明 | 🔓 可接受 | 仅修正文档 |

**推荐**: 方案B（短期），方案A（长期安全增强）

**理由**：
- Electron IPC 在可信边界内（同一应用的不同进程）
- 方案A 需要较大架构改动，当前任务不涉及
- 建议在架构文档中如实说明密钥传递机制

## 四、数据加密格式

### 4.1 当前格式（Base64编码）

```
salt[32] : iv[16] : authTag[16] : ciphertext[variable]
```

### 4.2 推荐格式（修正IV长度）

```
salt[32] : iv[12] : authTag[16] : ciphertext[variable]
```

### 4.3 格式版本标识（可选增强）

为支持渐进式迁移，可引入版本标识：

```typescript
interface EncryptedDataV2 {
  version: 2           // 格式版本号
  salt: Uint8Array     // 32 bytes
  iv: Uint8Array       // 12 bytes (修正后)
  authTag: Uint8Array  // 16 bytes (GCM内置)
  ciphertext: Uint8Array
}
```

### 4.4 迁移策略

**阶段1**: 支持双格式解密
- 解密时检测 IV 长度自动适配
- 新加密使用 12 字节 IV

**阶段2**: 渐进式重新加密
- 用户编辑条目时自动使用新格式
- 可选：批量重新加密功能

## 六、剪贴板安全方案

### 6.1 现有实现分析

```typescript
// vault.ts - copyPassword
await navigator.clipboard.writeText(plaintext)
// 1.5秒后仅清除UI状态，剪贴板内容未清除
setTimeout(() => { copiedEntryId.value = null }, 1500)
```

### 6.2 风险分析

| 风险 | 来源 | 等级 |
|------|------|------|
| 剪贴板历史记录 | Windows Clipboard History | 🟡 中 |
| 云同步剪贴板 | iCloud/iCloud Clipboard | 🔴 高 |
| 第三方应用读取 | 剪贴板管理器、恶意软件 | 🟡 中 |

### 6.3 安全措施建议

#### 必须实施

**剪贴板自动清除**：
```typescript
async function copyPassword(entryId: string): Promise<void> {
  // ... 解密密码 ...
  
  await navigator.clipboard.writeText(plaintext)
  copiedEntryId.value = entryId
  
  // ⚠️ 新增：30秒后清除剪贴板
  setTimeout(async () => {
    try {
      await navigator.clipboard.writeText('')
      copiedEntryId.value = null
    } catch {
      copiedEntryId.value = null
    }
  }, 30000)  // 30秒
}
```

**用户提示**：
- Toast消息："密码已复制，将在30秒后从剪贴板清除"
- 可在VaultPage或全局Toast组件实现

#### 可选增强

| 措施 | 描述 |
|------|------|
| 剪贴板历史警告 | 检测Windows Clipboard History是否启用，显示警告 |

## 八、安全边界定义

### 8.1 可信边界

| 边界 | 描述 |
|------|------|
| **主进程** | 可信任区，加密/解密操作在此执行 |
| **渲染进程** | 可信任区（同一应用），存储临时密钥 |
| **IPC通道** | 可信任通道（Electron内部通信） |
| **操作系统剪贴板** | ⚠️ 半信任区，数据可能被其他应用读取 |
| **文件系统** | ⚠️ 半信任区，数据可能被其他应用访问 |

### 8.2 安全假设

1. 用户设备未被恶意软件控制
2. 用户操作系统安全机制正常工作
3. 用户对物理访问设备有控制权

### 8.3 不在安全边界内

- 云同步剪贴板（iCloud等）
- 操作系统剪贴板历史
- 第三方剪贴板管理器
- 屏幕录制/截图
