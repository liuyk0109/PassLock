## 🔴 严重问题（必须修复）

### 1. 数据库非原子写入：崩溃时可能导致永久数据丢失

**位置**：[electron/main.ts#L99-L104](/d:/code/PassLock/electron/main.ts)

**问题描述**：

主密码修改 handler 中，步骤4 `await setEntries(entries)` 和步骤5 `await setMasterKeyVerify(newVerifyData)` 是两次独立的 `saveDatabase()` 调用。`setEntries` 成功后，磁盘上的数据库处于不一致中间态——条目已用新密码加密，但验证数据仍为旧密码。如果应用在两次写入之间崩溃（断电、OOM、进程被杀等），用户将**永久无法解锁密码库**：

- 旧密码：匹配验证数据，但无法解密条目
- 新密码：能解密条目，但不匹配验证数据

同样，回滚路径（catch 块）也执行了两次独立的 `saveDatabase()`，存在同样的风险。

对于密码管理器而言，任何可能导致永久数据丢失的场景都是不可接受的，即使窗口很小。

**修复方案**：

将条目和验证数据的更新合并为单次数据库写入，同时修改回滚路径：

```typescript
// 步骤4+5：合并为单次写入，确保原子性
const database = getDatabase()
database.data.entries = entries
const newVerifyData = createVerifyData(newPassword)
database.data.masterKeyVerify = newVerifyData
await database.write()

// 回滚路径也合并为单次写入
if (backup) {
  try {
    const db = getDatabase()
    db.data.entries = backup.entries
    db.data.masterKeyVerify = backup.masterKeyVerify!
    await db.write()
  } catch (rollbackError) {
    console.error('Rollback failed:', rollbackError)
  }
}
```

需要在 `main.ts` 顶部增加导入：`import { ..., getDatabase } from './database'`。

**影响**：严重 - 崩溃时可能导致用户永久丢失所有密码数据

## 🟡 警告（应该修复）

### 1. `removeAllListeners` 会清除通道上所有监听器

**位置**：[electron/preload.ts#L64-L76](/d:/code/PassLock/electron/preload.ts)

**问题描述**：

`ipcRenderer.removeAllListeners(progressChannel)` 会移除 `'change-password-progress'` 通道上的**全部**监听器，而非仅移除本次调用注册的那个。这存在两个隐患：

1. 若 `onProgress` 未传入（不注册 handler），`.finally()` 仍会调用 `removeAllListeners`，可能误删其他来源注册的监听器
2. 若未来有多个调用方同时使用此通道，`removeAllListeners` 会互相干扰

**修复方案**：

保存 handler 引用，使用 `removeListener` 精确移除：

```typescript
changeMasterPassword: (
  currentPassword: string,
  newPassword: string,
  onProgress?: (current: number, total: number) => void
): Promise<ChangeMasterPasswordResult> => {
  const progressChannel = 'change-password-progress'
  let handler: ((_event: any, current: number, total: number) => void) | null = null

  if (onProgress) {
    handler = (_event: any, current: number, total: number) => {
      onProgress(current, total)
    }
    ipcRenderer.on(progressChannel, handler)
  }

  return ipcRenderer.invoke('crypto:changeMasterPassword', currentPassword, newPassword)
    .finally(() => {
      if (handler) {
        ipcRenderer.removeListener(progressChannel, handler)
      }
    })
},
```

**影响**：中等 - 可能误删其他监听器，影响未来扩展性

### 2. 重加密循环直接修改内存中的活数据库引用

**位置**：[electron/main.ts#L89-L97](/d:/code/PassLock/electron/main.ts)

**问题描述**：

`getEntries()` 返回的是 `database.data.entries` 的**直接引用**（非副本），重加密循环 `entries[i].password = encrypt(...)` 直接修改了内存中的数据库对象。虽然当前回滚机制能恢复数据，但这种就地修改模式带来以下风险：

1. 循环部分失败时（如第50条解密抛异常），内存中的 `database.data.entries` 处于混合加密状态（0-49条新密码，50+条旧密码）。虽然磁盘未被写入，但若此时有其他 IPC handler 读取 `getEntries()`，会获取到不一致的数据。
2. 回滚路径中 `setEntries(backup.entries)` 和 `setMasterKeyVerify(backup.masterKeyVerify!)` 同样是两次非原子写入（同严重问题 #1 所述）。

**修复方案**：

在副本上执行重加密，成功后再提交到数据库：

```typescript
// 在副本上操作，不污染内存中的活数据库
const originalEntries = getEntries()
const reEncryptedEntries: VaultEntry[] = JSON.parse(JSON.stringify(originalEntries))

for (let i = 0; i < reEncryptedEntries.length; i++) {
  let plaintext = decrypt(originalEntries[i].password, currentPassword)
  reEncryptedEntries[i].password = encrypt(plaintext, newPassword)
  plaintext = ''
  event.sender.send('change-password-progress', i + 1, reEncryptedEntries.length)
}

// 合并写入（与严重问题 #1 的修复一致）
const database = getDatabase()
database.data.entries = reEncryptedEntries
const newVerifyData = createVerifyData(newPassword)
database.data.masterKeyVerify = newVerifyData
await database.write()
```

**影响**：中等 - 部分失败时内存数据不一致，可能影响并发 IPC 读取
