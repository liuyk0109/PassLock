## 严重问题（必须修复）

### 1. 导入未验证备份主密码与当前密码库主密码一致，可导致导入条目永久不可解密

**涉及文件**：`electron/main.ts#L247-L269`、`src/stores/vault.ts#L573-L627`

**问题**：`backup:validateBackup` 仅验证用户输入的密码是否匹配**备份文件**的 `masterKeyVerify`，但**不验证**该密码是否同时匹配**当前密码库**的 `masterKeyVerify`。这允许以下数据损坏场景：

1. 用户用主密码 "A" 解锁当前密码库
2. 用户导入一个用主密码 "B" 创建的备份文件
3. 导入验证弹窗中输入 "B" → `validateBackup` 验证通过（因为 "B" 匹配备份的验证数据）
4. 备份条目以其原始加密状态（用 "B" 加密）导入到当前密码库
5. 用户尝试查看/复制导入条目的密码时，`decrypt(entry.password, masterKey="A")` 将失败（AES-256-GCM 认证标签校验失败），条目**永久不可解密**

UI 文案也存在矛盾：`verify-description` 写的是"验证当前主密码"，而 `verify-warning` 写的是"密码必须与备份文件的主密码一致"，只有在两者相同时才自洽，但代码没有任何强制保证。

**修复建议**：在 `validateBackup` 中增加对当前密码库 `masterKeyVerify` 的双重验证：

```typescript
ipcMain.handle('backup:validateBackup', async (_event, filePath: string, password: string) => {
  try {
    const content = await fs.readFile(filePath, 'utf-8')
    const backupData: BackupData = JSON.parse(content)

    if (!backupData.version || !backupData.masterKeyVerify) {
      return { success: true, isValid: false, error: '备份文件格式不正确' }
    }

    // 验证密码匹配备份文件
    const isValidForBackup = verifyPassword(backupData.masterKeyVerify, password)
    if (!isValidForBackup) {
      return { success: true, isValid: false, error: '密码不匹配备份文件' }
    }

    // 新增：验证密码同时匹配当前密码库
    const currentVerifyData = getMasterKeyVerify()
    if (currentVerifyData) {
      const isValidForCurrent = verifyPassword(currentVerifyData, password)
      if (!isValidForCurrent) {
        return { success: true, isValid: false, error: '备份文件的主密码与当前密码库主密码不一致，无法导入' }
      }
    }

    return { success: true, isValid: true }
  } catch (error: any) {
    return { success: false, isValid: false, error: error.message || '验证失败' }
  }
})
```

### 2. `importData` 未验证备份文件格式，可处理恶意/损坏文件

**涉及文件**：`electron/main.ts#L350-L427`

**问题**：`backup:importData` 处理器直接读取文件并遍历 `backupData.entries`，完全不执行任何格式验证（无 `version` 检查、无 `masterKeyVerify` 检查、无 `entries` 数组存在性检查）。虽然正常流程中 `parseBackup` 和 `validateBackup` 会先执行，但 `importData` 本身作为独立 IPC handler 无任何防御。

如果渲染进程被攻击或流程被绕过，直接调用 `backup:importData` 传入一个不含 `entries` 字段的文件，`for (const entry of backupData.entries)` 将在 `undefined` 上迭代，抛出未处理的异常。更严重的情况，恶意构造的 JSON 可注入非预期字段到数据库中。

**修复建议**：在 `importData` 开头添加格式验证：

```typescript
// 在读取 backupData 后添加
if (typeof backupData.version !== 'number' || backupData.version > 1) {
  return { success: false, error: '无效的备份文件格式', stats: { added: 0, overridden: 0, skipped: 0, keptBoth: 0 } }
}
if (!Array.isArray(backupData.entries)) {
  return { success: false, error: '备份文件数据格式错误', stats: { added: 0, overridden: 0, skipped: 0, keptBoth: 0 } }
}
```

## 警告（应该修复）

### 1. 导入操作非原子性——逐条写入数据库，崩溃可导致数据不一致

**涉及文件**：`electron/main.ts#L378-L399`

**问题**：`importData` 中对每条备份条目分别调用 `await addEntry(entry)` 和 `await updateEntry(entry.id, entry)`，每次调用都触发一次完整的数据库文件写入（`database.write()`）。对比 `changeMasterPassword` 采用了"内存修改→单次写入"的原子模式，`importData` 的非原子写入存在以下风险：

- 如果处理到第 N 条时进程崩溃（断电、OS 杀进程），已写入的前 N-1 条无法回滚
- 回滚机制仅在 catch 中生效，无法处理进程级崩溃

**修复方向**：参照 `changeMasterPassword` 的模式，在内存中完成所有条目修改后，执行单次 `database.write()`：

```typescript
// 在内存中修改 database.data.entries
const database = getDatabase()
for (const entry of backupData.entries) {
  if (localIdSet.has(entry.id)) {
    const action = resolutionMap.get(entry.id) ?? 'skip'
    if (action === 'override') {
      const index = database.data.entries.findIndex(e => e.id === entry.id)
      if (index !== -1) database.data.entries[index] = { ...entry, updatedAt: Date.now() }
      overridden++
    } else if (action === 'keepboth') {
      database.data.entries.push({ ...entry, id: crypto.randomUUID() })
      keptBoth++
    } else { skipped++ }
  } else {
    database.data.entries.push(entry)
    added++
  }
}
await database.write() // 单次原子写入
```

### 2. 冲突处理弹窗中"更改"按钮无效——无法撤销已选择的冲突解决方案

**涉及文件**：`src/components/ImportConflictDialog.vue#L143-L149`

**问题**：当用户选择了某个冲突解决方案后，界面显示"已选择：覆盖/跳过/保留两者"及一个"更改"按钮。但该按钮的点击事件为：

```html
<button class="change-btn" @click="handleSetResolution(conflict.id, getResolution(conflict.id)!)">更改</button>
```

这会以**相同的 action** 再次调用 `setConflictResolution`，不会清除已选方案，因此 `v-if="getResolution(conflict.id)"` 始终为 truthy，操作按钮永远不会重新显示。用户一旦做出选择就**无法更改**。

**修复建议**：在 store 中添加 `removeConflictResolution` 方法，"更改"按钮调用它来清除选择：

```typescript
// vault.ts 新增
function removeConflictResolution(entryId: string): void {
  conflictResolutions.value.delete(entryId)
  conflictResolutions.value = new Map(conflictResolutions.value)
}

// ImportConflictDialog.vue 修改
<button class="change-btn" @click="emit('remove-resolution', conflict.id)">更改</button>
```

同时在 Emits 中新增 `(e: 'remove-resolution', entryId: string): void`。

### 3. `dialog:showSaveDialog/showOpenDialog` 参数类型为 `any`，存在安全风险

**涉及文件**：`electron/preload.ts#L159-L164`

**问题**：preload 中 dialog API 的 options 参数类型为 `any`：

```typescript
dialog: {
  showSaveDialog: (options?: any) =>
    ipcRenderer.invoke('dialog:showSaveDialog', options),
  showOpenDialog: (options?: any) =>
    ipcRenderer.invoke('dialog:showOpenDialog', options),
},
```

而主进程接收端使用 `Electron.SaveDialogOptions` / `Electron.OpenDialogOptions` 完整类型，渲染进程可传入不受限的选项（如危险 `properties`）。对于密码管理器应更加严格。

**修复建议**：在 preload 中使用与 `electron.d.ts` 一致的受限类型：

```typescript
dialog: {
  showSaveDialog: (options?: SaveDialogOptions) =>
    ipcRenderer.invoke('dialog:showSaveDialog', options),
  showOpenDialog: (options?: OpenDialogOptions) =>
    ipcRenderer.invoke('dialog:showOpenDialog', options),
},
```

同时在主进程 handler 中对传入 options 进行字段白名单过滤，仅允许 `title`、`defaultPath`、`filters`、`properties` 等安全字段。
